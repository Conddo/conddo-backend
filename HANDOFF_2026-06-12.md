## 1. Decision summary

- **2 platform roles stay**: `TENANT_ADMIN` (the owner) and `STAFF`. `SUPER_ADMIN` is Conddo ops.
- **5 new STAFF sub-roles** distinguish what a staffer does. STAFF users carry a `staff_role` enum column alongside `role`. TENANT_ADMIN users do not (their `staff_role` is NULL).
- **Owners use `/dashboard`** (the existing FE shell). **Staff use `/work/*`** ‚Äî a lean shell whose sidebar is scoped to a single role.

### Sub-role enum (FE source of truth lives in `lib/api/staff.ts ‚Üí STAFF_ROLE_CATALOGUE`)

| Key | Label | What they do |
|---|---|---|
| `MANAGER` | Manager | Second-in-command. Everything except billing + staff invites. |
| `PHARMACIST` | Pharmacist | Clinical access ‚Äî EMR, prescriptions, consultations, follow-ups. |
| `CASHIER` | Cashier | POS sales + read-only customers. No revenue, no inventory writes. |
| `STOCK_MANAGER` | Stock Manager | Inventory, restock, reconciliation, bulk upload, movement log. |
| `BOOKKEEPER` | Bookkeeper | Read-only orders + payments + analytics. CSV exports. |

Full per-role access matrix in the FE catalogue. Treat that file as the canonical reference; mirror its `access` strings in the welcome email body if you can.

## 2. Migration

V47 (or next available):

```sql
ALTER TABLE tenant_users
  ADD COLUMN staff_role VARCHAR(20);

-- Backfill: every existing STAFF user defaults to MANAGER (closest to
-- "they could do almost everything"). Owners stay NULL.
UPDATE tenant_users
   SET staff_role = 'MANAGER'
 WHERE role = 'STAFF';

ALTER TABLE tenant_users
  ADD CONSTRAINT tenant_users_staff_role_check
  CHECK (staff_role IS NULL OR staff_role IN
    ('MANAGER','PHARMACIST','CASHIER','STOCK_MANAGER','BOOKKEEPER'));

-- Pair invariant ‚Äî owners never carry a sub-role; staff always do
-- after we backfill.
ALTER TABLE tenant_users
  ADD CONSTRAINT tenant_users_staff_role_pair
  CHECK (
    (role = 'TENANT_ADMIN' AND staff_role IS NULL) OR
    (role = 'STAFF' AND staff_role IS NOT NULL) OR
    (role NOT IN ('TENANT_ADMIN', 'STAFF'))   -- SUPER_ADMIN, etc.
  );
```

## 3. `/me` shape ‚Äî add `staffRole`

The FE already reads `me.user.staffRole`. Surface it from the JWT subject's row:

```jsonc
// GET /api/v1/me
{
  "data": {
    "user": {
      "id": "uuid",
      "fullName": "...",
      "email": "...",
      "role": "TENANT_ADMIN" | "STAFF" | "SUPER_ADMIN",
      "staffRole": "MANAGER" | "PHARMACIST" | "CASHIER" | "STOCK_MANAGER" | "BOOKKEEPER" | null,
      "initials": "TB"
    },
    "tenant": { /* unchanged */ }
  }
}
```

JWT claims should also include `staffRole` so `@PreAuthorize` expressions can check it without a DB hit. Suggested addition to the JwtClaims shape next to the existing `role`.

## 4. `@PreAuthorize` updates

Today most controllers use `READ = hasAnyRole('TENANT_ADMIN','STAFF','SUPER_ADMIN')` and `WRITE = hasAnyRole('TENANT_ADMIN','SUPER_ADMIN')`. That stays for **Manager** (effectively current STAFF). For the other 4 sub-roles, gate by `staffRole` claim:

```java
// Example ‚Äî InventoryController. Stock Manager + Manager + Owner can write;
// Cashier, Pharmacist, Bookkeeper get read-only.
@PreAuthorize("@staffAccess.canRead('inventory')")
@PreAuthorize("@staffAccess.canWrite('inventory')")
```

Recommend implementing a small bean `StaffAccess` (similar to `FeatureFlagGuard`) that maps `(staffRole, module)` ‚Üí `read | write | none`. The mapping lives BE-side; FE just respects 403s. Module keys to support: `inventory`, `orders`, `payments`, `customers`, `analytics`, `prescriptions`, `consultations`, `pos`, `emr`, `staff`, `billing`, `marketing`.

If you prefer, paste the FE catalogue's `access` strings into a static table and machine-derive the matrix from that ‚Äî single source of truth.

### Special cases

- **Billing** (`/api/v1/billing/*` writes): Owner-only as today.
- **Staff** (`/api/v1/staff/*` writes): Owner-only.
- **POS void on completed sales**: Owner-only (POS handoff ¬ß3 already calls this out).
- **EMR notes write**: PHARMACIST + MANAGER + Owner. Per Beta 4 handoff ¬ß4.
- **Discount approval**: Owner-only (Beta 12B already does this).

## 5. Invite acceptance flow

Two new endpoints. Both live under `/auth/*` (no `/api/v1` prefix, no JWT required ‚Äî these are pre-auth).

### `GET /auth/invite/preview?token=<token>`

Returns a small preview so the FE can render *"You've been invited to **Wellspring Pharmacy** as a **Cashier**"* before asking for password.

```jsonc
// 200
{
  "data": {
    "tenantName": "Wellspring Pharmacy",
    "roleLabel": "Cashier",
    "staffRole": "CASHIER",
    "email": "tunde@gmail.com",
    "invitedBy": "Mrs Adebayo" | null
  }
}

// 404 ‚Äî token invalid / expired / consumed
// FE silently treats any non-200 as "invite not active"
```

### `POST /auth/accept-invite`

```jsonc
// request
{
  "token": "single-use-token",
  "password": "8+ chars",
  "fullName": "Tunde Bello"          // optional ‚Äî overrides BE pre-fill
}

// 200 ‚Äî same LoginResult shape the normal /auth/login returns
{
  "data": {
    "accessToken": "...",
    "tokenType": "Bearer",
    "expiresIn": 900,
    "userId": "uuid",
    "role": "STAFF"
  }
}
```

Errors the FE handles by name:
- `INVITE_EXPIRED` ‚Äî token TTL elapsed
- `INVITE_INVALID` ‚Äî token doesn't exist / already used / malformed
- standard `VALIDATION_ERROR` for short passwords (we pre-check ‚â• 8 chars, defence-in-depth)

On success, flip the user's `status` from `invited` ‚Üí `active`, set the bcrypt'd password, store fullName if provided, issue the access token. FE auto-routes to `/work/<role-landing>` based on `me.user.staffRole`.

### `POST /api/v1/staff/invite` ‚Äî updated body

Add `staffRole` (was `role` which used `STAFF` / `TENANT_ADMIN`):

```jsonc
{
  "email": "tunde@gmail.com",
  "staffRole": "CASHIER",
  "fullName": "Tunde Bello"      // optional
}
```

BE behaviour:
1. Generate a single-use `acceptInviteToken` (signed, TTL 72h).
2. Create the `tenant_users` row with `role = 'STAFF'`, `staff_role = <body.staffRole>`, `status = 'invited'`, no password hash yet.
3. Send the invite email (Brevo) ‚Äî template: "Tunde, you've been invited to join Wellspring Pharmacy as a Cashier. Click here to accept: {{appUrl}}/accept-invite?token={{token}}". Include a 5-bullet summary of role access for context.
4. Return the new StaffMember row (no token in body ‚Äî only via email).

Plan-gated:
- Launcher: max 2 active staffers (owner + 1)
- Growth: max 5 active staffers (owner + 4)
- Scaler: unlimited

Reject with `409 PLAN_LIMIT_REACHED` when the limit is hit so the FE can surface the existing upgrade-prompt copy.

### `PATCH /api/v1/staff/{id}` ‚Äî updated body

Was `{ role, active }`. Now `{ staffRole, active }`:

```jsonc
{
  "staffRole": "PHARMACIST",
  "active": true
}
```

Refuse with `403 OWNER_PROTECTED` if the target is the owner (role = TENANT_ADMIN). FE already gates the UI but defence-in-depth.

## 6. Email templates (Brevo)

Two new templates. Pass `{{firstName}}`, `{{tenantName}}`, `{{roleLabel}}`, `{{roleAccessLines}}` (the bullets), and `{{acceptUrl}}` so the FE catalogue stays the source of truth.

1. **Invite email** ‚Äî sent on `POST /staff/invite`. Subject: *"{{tenantName}} invited you to join as {{roleLabel}}"*. Body explains the role's access in plain English; CTA = Accept invite.
2. **Welcome email** ‚Äî sent on `POST /auth/accept-invite`. Subject: *"You're in ‚Äî welcome to {{tenantName}}"*. Confirms the account is active, links to their work surface.

## 7. Suggested shipping order

1. **Migration + `/me` shape** (one PR, low risk ‚Äî backfill defaults STAFF ‚Üí MANAGER which preserves current behaviour exactly).
2. **`@PreAuthorize` + `StaffAccess` bean** ‚Äî the bulk of the BE work. Wire it module by module; FE breaks gracefully if a module returns 403 (PlanGate already handles it).
3. **Invite acceptance endpoints + email template**. Smallest BE surface but biggest UX uplift ‚Äî without it, current STAFF onboarding is rough (today they sign in with a BE-stored temp password).
4. **Plan-gated staff count** ‚Äî last; easy to add after roles are in place.
