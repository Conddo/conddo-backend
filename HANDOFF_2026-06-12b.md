## 1. Why this is needed

Currently, when a tenant clicks "Request Beta access" on `/features` (in `conddo-app`), BE records the row:

```
tenant_feature_flags { tenant_id, feature_key, interest=true, enabled=false, interest_at=now() }
```

Ops can grant per-tenant via the existing endpoint:

```
POST /api/v1/admin/tenants/{tenantId}/feature-flags/{featureKey}/grant
```

â€¦but there's **no way to query the queue**. Ops doesn't know which tenants are waiting unless they tail the BE logs. The Studio admin page fixes that.

## 2. Endpoints to ship on Studio backend

Studio backend already has JPA access to the shared Postgres' `tenant_feature_flags` table (same pattern Platform Tenants / Sites use). Add a controller at `/api/jobs/admin/platform/feature-flags` with these three endpoints:

### `GET /api/jobs/admin/platform/feature-flags`

```jsonc
// Query params (all optional)
?status=interest   // "interest" | "granted" | "revoked" | "all"   (default: "interest")
&featureKey=cashback_loyalty  // filter to one feature

// 200 response â€” array sorted by interest_at DESC (oldest pending first)
{
  "success": true,
  "data": [
    {
      "tenantId": "uuid",
      "featureKey": "cashback_loyalty",
      "tenantName": "Wellspring Pharmacy",
      "tenantSlug": "wellspring-pharmacy",
      "tenantVertical": "pharmacy",
      "tenantPlan": "growth",
      "status": "interest" | "granted" | "revoked",
      "interestAt": "2026-06-10T08:00:00Z",
      "grantedAt": null,
      "grantedByName": null
    },
    â€¦
  ]
}
```

**Status derivation** (server-side):
- `interest`: row has `interest=true` AND `enabled=false` AND `granted_at IS NULL`
- `granted`: row has `enabled=true`
- `revoked`: row has `enabled=false` AND `granted_at IS NOT NULL` (was granted, then walked back)

**Tenant joins** (`tenantName`, `tenantSlug`, `tenantVertical`, `tenantPlan`): server reads from `tenants` table. `grantedByName` joins through `tenant_users` (or `staff_members` if that's where SUPER_ADMIN names live â€” same path Platform Sites uses for `qaApprovedByName`).

**Auth:** `@PreAuthorize("hasRole('ADMIN')")` for write actions; reads can include `TEAM_LEAD` so a lead can see the queue without granting.

### `POST /api/jobs/admin/platform/feature-flags/{tenantId}/{featureKey}/grant`

Returns the same row shape, now with `status="granted"`, `grantedAt=now()`, `grantedByName=<acting staff>`. Internally calls the existing `TenantFeatureFlagService.grantAccess(tenantId, featureKey, grantedBy)` so behaviour is consistent with the conddo-backend admin endpoint.

```jsonc
// No body
// 200 â€” same FeatureFlagRequest shape
```

### `POST /api/jobs/admin/platform/feature-flags/{tenantId}/{featureKey}/revoke`

Calls `TenantFeatureFlagService.revokeAccess(tenantId, featureKey)`. Keeps `interest_at` and `granted_at` set so the row stays auditable; sets `enabled=false`.

```jsonc
// No body
// 200 â€” same FeatureFlagRequest shape
```

**Why not just hit conddo-backend's admin endpoints directly from Studio?** Studio JWTs are issued by Studio's `/api/jobs/auth/login` and carry the Studio role set (`ADMIN`/`TEAM_LEAD`/etc); conddo-backend's admin endpoints check `hasRole('SUPER_ADMIN')` which lives in the platform JWT. Bridging would need a cross-service token exchange. Easier: Studio backend has DB access already, so it just owns these three endpoints itself and shares the service layer.
