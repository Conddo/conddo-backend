## 2) Asks

### 2.1 — Fix `PATCH /api/v1/inventory/categories/{id}` (P0)

**Request body** (FE already sends):

```json
{ "name": "Antihypertensives" }
```

**Expected response** — same shape as `POST` create:

```json
{
  "success": true,
  "data": { "id": "<uuid>", "name": "Antihypertensives", "productCount": 12 }
}
```

**Validation:**
- 400 if `name` is missing / blank / > 80 chars
- 409 if another category in the same tenant already has that name (case-insensitive). The FE pre-checks but a concurrent rename could race — please return 409 not 500.
- 404 if id doesn't exist in tenant
- Otherwise 200 / updated row

### 2.2 — Fix `DELETE /api/v1/inventory/categories/{id}` (P0)

**Expected response:** `204 No Content` on success.

**Constraints:**
- 409 if any product references this category. FE will refuse to send the request when `productCount > 0`, but again — concurrent edits exist. Return 409 with a useful message so FE can toast it; don't 500.
- 404 if id doesn't exist in tenant
- 204 otherwise

We do **not** want a cascade delete or a "ghost" rename to "Uncategorised" — the user moves products manually first. If you want to offer a `?reassignTo={otherCategoryId}` query param later, it's additive and we'll wire it.

### 2.3 — Add `productCount` to `GET /inventory/categories` response (P1)

FE already declares the field as optional and renders `—` when missing. Once you ship it, the category page shows the count and the Delete button correctly enables/disables.

**Cheapest query path:** `SELECT c.id, c.name, COUNT(p.id) FROM categories c LEFT JOIN products p ON p.category_id = c.id WHERE c.tenant_id = :tenant GROUP BY c.id`. Index on `products.category_id` if it's not already there.

**Shape:**

```json
{
  "success": true,
  "data": [
    { "id": "uuid", "name": "Analgesics", "productCount": 12 },
    { "id": "uuid", "name": "Anti-Malaria", "productCount": 4 }
  ]
}
```

## 6) ALSO P0 — JWT `activeModules` / `vertical` / `plan` claims (the manifest-nav unblock)

The `Sessions` nav item for music-studio that we asked BE to add to the manifest registry **did ship** (commit `b4c7422`). But it doesn't appear in the FE sidebar because the FE never calls `GET /api/v1/registry/manifests` — and that's the root cause for every vertical-specific deep-dive (Sessions, Fittings, Fabric, Prescriptions, Consultations) being invisible right now.

### What the FE does today

In [conddo-app/hooks/useManifests.ts](conddo-app/hooks/useManifests.ts):

```ts
const modules = decodeJwt(getAccessToken())?.activeModules;
if (!modules || modules.length === 0) {
  setLoading(false);
  return;          // ← never calls /registry/manifests
}
api.get<UIManifest[]>(`/registry/manifests?modules=${encodeURIComponent(modules.join(","))}`) …
```

So as long as the access token JSON payload contains **no `activeModules` array**, the manifest registry is dead weight. The FE falls back to the hardcoded [lib/app-nav.ts](conddo-app/lib/app-nav.ts), which has no vertical-specific entries.

### Confirmation

The local FE memory note from 2026-05-24 already flagged this: *"JWT still lacks `activeModules`/`vertical`/`plan`."* A quick `decodeJwt` of any access token from `/auth/login` confirms it — `sub`, `tenantId`, `role`, `iat`, `exp` are present; `activeModules`, `vertical`, `plan` are not.

### The ask

Add three claims to the access token issued by `POST /auth/login` (and `POST /auth/refresh`) — exact spec from `conddo_architecture.md` §16:

```json
{
  "sub": "<userId>",
  "tenantId": "<uuid>",
  "role": "TENANT_ADMIN",
  "vertical": "music-studio",
  "plan": "growth",
  "activeModules": ["core.bookings", "music-studio.sessions", "core.customers", …],
  "iat": …,
  "exp": …
}
```

- **`vertical`** — the tenant's vertical id (matches `tenants.vertical_id`).
- **`plan`** — the active billing plan slug (`launcher` | `growth` | `scaler`).
- **`activeModules`** — array of tool/module ids the tenant currently has enabled, derived from `tenant.modules` (or the equivalent resolved set after applying plan + vertical defaults). This is the field `GET /registry/manifests?modules=…` already expects as input.

### What unblocks the moment this lands

The FE has zero changes to make once the claims appear. `useManifests` will see `activeModules` in the token, call `/registry/manifests`, render the BE-supplied tool sections, and the static APP_NAV fallback (plus the FE-side `VERTICAL_EXTRAS` stopgap I'm shipping today in [hooks/useAppNav.ts](conddo-app/hooks/useAppNav.ts)) goes dormant automatically. So:

- Music studio tenants see **Sessions** (the manifest entry you already shipped).
- Fashion tenants see **Fittings** and **Fabric**.
- Pharmacy tenants see **Prescriptions** and **Consultations**.
- Future verticals (food-and-beverage, beauty-and-wellness, logistics, professional-services) get a sidebar update with **zero FE work** — you just add their tool manifests to the registry.

### Stopgap I'm shipping in the meantime

`hooks/useAppNav.ts` now splices vertical-specific entries into APP_NAV based on `verticalOf(me)`, so music-studio tenants get Sessions, fashion gets Fittings + Fabric, pharmacy gets Prescriptions + Consultations — *today*, without waiting on BE. The branch that does this only fires when `manifests === null || sections.length === 0`, so the day BE ships the JWT claims, the stopgap stops contributing entirely and the manifest-driven nav takes over cleanly.

### Acceptance

```bash

echo "$TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | jq

# Expect to see "vertical", "plan", and a non-empty "activeModules" array.
```

## 7) Small follow-ups (P2 — non-blocking)

These two surfaced while verifying the §6 deploy. Neither breaks anything today — flagging for cleanup whenever you're back in those files.

### 7.1 — JWT claim key is `tenant_id` (snake_case)

Other access-token claims are camelCase / lowercase (`sub`, `vertical`, `plan`, `activeModules`, `iat`, `exp`, `iss`). The tenant id is the odd one out:

```json
{ "tenant_id": "3633805e-…", "sub": "…", "vertical": "music-studio", "activeModules": [...] }
```

FE doesn't actually read `tenantId` off the token today (it gets that from `/me`), so this is moot in production — but [conddo-app/lib/jwt.ts:7](conddo-app/lib/jwt.ts#L7) declares the field as `tenantId?` per the architecture spec, and we'd rather have one wire format. Rename to `tenantId` when convenient.

### 7.2 — Plan slug is `"starter"`, should be `"launcher"`

The login response decodes to `"plan": "starter"` for a tenant created with `planId: null`. We renamed the tiers a few weeks back — the pricing copy throughout the FE (and the BE's own [PRICING_TIERS](backend/conddo-pricing-tiers.md) doc) now uses **launcher / growth / scaler**. The JWT default fell behind.

Either the unsubscribed default should be `"launcher"` (matching the free/lowest tier name) or, more correctly, omit the `plan` claim entirely until the tenant subscribes — both FE and BE plan-gate checks already handle "no plan" as "treat as launcher". The current behavior is harmless because nothing in FE actually keys off the string `"starter"`, but it's a confusing breadcrumb if someone debugs a token later.
