## 2) Asks

### 2.1 ‚Äî Fix `PATCH /api/v1/inventory/categories/{id}` (P0)

**Request body** (FE already sends):

```json
{ "name": "Antihypertensives" }
```

**Expected response** ‚Äî same shape as `POST` create:

```json
{
  "success": true,
  "data": { "id": "<uuid>", "name": "Antihypertensives", "productCount": 12 }
}
```

**Validation:**
- 400 if `name` is missing / blank / > 80 chars
- 409 if another category in the same tenant already has that name (case-insensitive). The FE pre-checks but a concurrent rename could race ‚Äî please return 409 not 500.
- 404 if id doesn't exist in tenant
- Otherwise 200 / updated row

### 2.2 ‚Äî Fix `DELETE /api/v1/inventory/categories/{id}` (P0)

**Expected response:** `204 No Content` on success.

**Constraints:**
- 409 if any product references this category. FE will refuse to send the request when `productCount > 0`, but again ‚Äî concurrent edits exist. Return 409 with a useful message so FE can toast it; don't 500.
- 404 if id doesn't exist in tenant
- 204 otherwise

We do **not** want a cascade delete or a "ghost" rename to "Uncategorised" ‚Äî the user moves products manually first. If you want to offer a `?reassignTo={otherCategoryId}` query param later, it's additive and we'll wire it.

### 2.3 ‚Äî Add `productCount` to `GET /inventory/categories` response (P1)

FE already declares the field as optional and renders `‚Äî` when missing. Once you ship it, the category page shows the count and the Delete button correctly enables/disables.

**Cheapest query path:** `SELECT c.id, c.name, COUNT(p.id) FROM categories c LEFT JOIN products p ON p.category_id = c.id WHERE c.tenant_id = :tenant GROUP BY c.id`. Index on `products.category_id` if it's not already there.

**Shape:**

```json
{
  "success": true,
  "data": [
    { "id": "uuid", "name": "Analgesics", "productCount": 12 },
    { "id": "uuid", "name": "Anti-Malaria", "productCount": 4 }
  ]
}
```

## 6) ALSO P0 ‚Äî JWT `activeModules` / `vertical` / `plan` claims (the manifest-nav unblock)

The `Sessions` nav item for music-studio that we asked BE to add to the manifest registry **did ship** (commit `b4c7422`). But it doesn't appear in the FE sidebar because the FE never calls `GET /api/v1/registry/manifests` ‚Äî and that's the root cause for every vertical-specific deep-dive (Sessions, Fittings, Fabric, Prescriptions, Consultations) being invisible right now.

### What the FE does today

In [conddo-app/hooks/useManifests.ts](conddo-app/hooks/useManifests.ts):

```ts
const modules = decodeJwt(getAccessToken())?.activeModules;
if (!modules || modules.length === 0) {
  setLoading(false);
  return;          // ‚Üê never calls /registry/manifests
}
api.get<UIManifest[]>(`/registry/manifests?modules=${encodeURIComponent(modules.join(","))}`) ‚Ä¶
```

So as long as the access token JSON payload contains **no `activeModules` array**, the manifest registry is dead weight. The FE falls back to the hardcoded [lib/app-nav.ts](conddo-app/lib/app-nav.ts), which has no vertical-specific entries.

### Confirmation

The local FE memory note from 2026-05-24 already flagged this: *"JWT still lacks `activeModules`/`vertical`/`plan`."* A quick `decodeJwt` of any access token from `/auth/login` confirms it ‚Äî `sub`, `tenantId`, `role`, `iat`, `exp` are present; `activeModules`, `vertical`, `plan` are not.

### The ask

Add three claims to the access token issued by `POST /auth/login` (and `POST /auth/refresh`) ‚Äî exact spec from `conddo_architecture.md` ¬ß16:

```json
{
  "sub": "<userId>",
  "tenantId": "<uuid>",
  "role": "TENANT_ADMIN",
  "vertical": "music-studio",
  "plan": "growth",
  "activeModules": ["core.bookings", "music-studio.sessions", "core.customers", ‚Ä¶],
  "iat": ‚Ä¶,
  "exp": ‚Ä¶
}
```

- **`vertical`** ‚Äî the tenant's vertical id (matches `tenants.vertical_id`).
- **`plan`** ‚Äî the active billing plan slug (`launcher` | `growth` | `scaler`).
- **`activeModules`** ‚Äî array of tool/module ids the tenant currently has enabled, derived from `tenant.modules` (or the equivalent resolved set after applying plan + vertical defaults). This is the field `GET /registry/manifests?modules=‚Ä¶` already expects as input.

### What unblocks the moment this lands

The FE has zero changes to make once the claims appear. `useManifests` will see `activeModules` in the token, call `/registry/manifests`, render the BE-supplied tool sections, and the static APP_NAV fallback (plus the FE-side `VERTICAL_EXTRAS` stopgap I'm shipping today in [hooks/useAppNav.ts](conddo-app/hooks/useAppNav.ts)) goes dormant automatically. So:

- Music studio tenants see **Sessions** (the manifest entry you already shipped).
- Fashion tenants see **Fittings** and **Fabric**.
- Pharmacy tenants see **Prescriptions** and **Consultations**.
- Future verticals (food-and-beverage, beauty-and-wellness, logistics, professional-services) get a sidebar update with **zero FE work** ‚Äî you just add their tool manifests to the registry.

### Stopgap I'm shipping in the meantime

`hooks/useAppNav.ts` now splices vertical-specific entries into APP_NAV based on `verticalOf(me)`, so music-studio tenants get Sessions, fashion gets Fittings + Fabric, pharmacy gets Prescriptions + Consultations ‚Äî *today*, without waiting on BE. The branch that does this only fires when `manifests === null || sections.length === 0`, so the day BE ships the JWT claims, the stopgap stops contributing entirely and the manifest-driven nav takes over cleanly.
