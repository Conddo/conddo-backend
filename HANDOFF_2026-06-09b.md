## 2) What I've confirmed

| Probe | Result |
|---|---|
| Unauthed `GET /api/v1/inventory/movements` | **401** — routes ARE registered |
| Authed `GET /api/v1/inventory/categories` (same tenant + same token) | **200** with `{success:true,data:[]}` — auth + token are good |
| Authed `GET /api/v1/inventory/movements` | **500 with empty body** |
| Authed `POST /api/v1/inventory/restock` | **500 with empty body** |
| Authed `POST /api/v1/inventory/adjustment` | **500 with empty body** |
| Authed `POST /api/v1/inventory/reconciliations` | **500 with empty body** |

The empty 500 (no `{success:false,error:{...}}` envelope) is the key tell. The global exception handler that wraps every other 5xx in the standard envelope isn't running for these endpoints. That suggests the failure happens **before** the controller method body (or in early initialization), so the handler never gets a chance.

## 3) Likely root cause — short list

In rough order of probability:

1. **V34 migration didn't apply on Render.** Flyway `pharmacy_stock_movements` / `pharmacy_reconciliations` / `pharmacy_reconciliation_items` tables don't exist in prod, so the JPA query throws `PSQLException: relation does not exist` during repository init. Check `flyway_schema_history` on the prod DB.
2. **`StockMovementService` bean wiring crash.** The `StockEventPublisher` is `@Autowired(required = false)` — but if any constructor downstream needs it as `@NotNull`, the service bean fails to construct and every method invocation throws. Check the Render startup log around `BeanCreationException`.
3. **Redis client config issue** despite the `required = false` defensive wrapping. The `connect-timeout=200ms` test pin you mention in the commit message may not apply in prod, so a slow/missing Redis blocks the publisher init.
4. **`@PreAuthorize(READ)` / `WRITE` constants are pharmacy-vertical-gated** but the verticalId on this tenant is `pharmacy` (confirmed via JWT decode), so this is unlikely — but worth a glance if your `READ`/`WRITE` SpEL expressions reference a different claim.

## 5) Tiny doc nit

The spec doc (`conddo-pharmacy-api-spec-v2.md` §12A) shows paths like `/dashboard/{slug}/pharmacy/inventory/movements`. The controller actually ships them under `/api/v1/inventory/*` (tenant-from-JWT, no slug in URL). The controller path is what FE wires to — it's consistent with the rest of the inventory surface — but consider updating the spec doc so the next reader doesn't waste time hunting the dashboard-slug variant. Same applies to the `reconciliation/{id}/count` (singular) vs the shipped `/counts` (plural), and `POST /reconciliations` vs the spec's implied `POST /reconciliation/start`.
